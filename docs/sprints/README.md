## Usage

- Treat each sprint doc as the source of truth for that phase.
- Do not begin a sprint until its dependencies are completed or explicitly waived.
- Update the sprint doc itself as decisions change so the plan stays operational rather than archival.

## Sequence

1. Sprint 0 establishes the environment and repository baseline.
2. Sprint 1 proves the PDF can be parsed into a stable, normalized model.
3. Sprint 2 locks the backend contract and persistence model.
4. Sprint 3 builds the modern Flutter client foundation.
5. Sprint 4 ships the section-first student experience.
6. Sprint 5 adds reminders and change notifications safely.
7. Sprint 6 operationalizes timetable import and rollback.
8. Sprint 7 hardens the system for beta usage.
9. Sprint 8 tunes cost, scale, and operational headroom for roughly 2,000 users.
